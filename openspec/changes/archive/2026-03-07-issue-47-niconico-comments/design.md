# design: ニコ生風コメント表示機能

## Context

現在のゲームは Phaser 3 ベースのキャンバス描画と HTML オーバーレイ（HUD・タイトル・リザルト）で構成されている。
ゲームプレイ中の演出はすべて Phaser のキャンバス内で行われており、コメントも同様にキャンバス内 Text オブジェクトとして実装する。

コメントデータは将来 Azure Blob Storage からの動的読み込みに差し替えられることを考慮し、`fetch` ベースの設計とする。
今回のスコープでは `public/comments.json`（静的ファイル）を使用し、URL を環境変数で上書きできる余地を残す。

## Goals / Non-Goals

**Goals:**

- ゲームプレイ中にニコ生風コメントを画面右→左へ流す
- 長いコメントほど速く流れる（ニコ生方式：全コメントが同一の画面横断時間）
- 6 レーン構成でコメント同士の重なりを防ぐ
- ゲームイベント（転倒・腰痛・ゴール）発生時にコメントをバースト投入する
- HUD の HTML ボタンで ON/OFF を切り替えられる
- コメントデータをコードと分離し、再デプロイなしに差し替え可能にする

**Non-Goals:**

- SNS・X (Twitter) との連携（別 change にて検討）
- Azure Blob Storage からの動的読み込み（別 change にて検討）
- ユーザーによるコメント投稿機能
- コメントのアニメーション効果（フェードイン等）

## Decisions

### D1: コメントを Phaser Text で描画する

**採用**: `Phaser.GameObjects.Text` を使用し、キャンバス内に描画する。
**理由**: ゲームの他オブジェクトと depth で重なり順を制御しやすく、`scene.add.text()` で DotGothic16 フォントをそのまま使用できる。
**代替案**: HTML オーバーレイ（CSS アニメーション）— ゲームの depth 管理外になるため却下。

### D2: 速度計算はニコ生方式（全コメント同一横断時間）

**採用**: 全コメントが `CROSSING_DURATION`（固定値、例: 6000ms）で画面を横断するよう速度を計算する。

```
speed = (CANVAS_W + text.width) / CROSSING_DURATION  [px/ms]
```

**理由**: 長いコメントほど自然に速くなる。ゲームのスクロール速度と独立した演出次元を保てる。
**代替案**: 固定速度 — 長いコメントがレーンを長時間占有し詰まりやすいため却下。

### D3: 6 レーン構成、画面上部に配置

**採用**: Y = 25, 55, 85, 115, 145, 175 の 6 レーン。
**理由**: プレイヤーやゲームオブジェクトは画面中〜下部に集中しているため、上部にコメントを配置することでプレイの邪魔にならない。
レーン選択は「前コメントの末尾が画面右端から 30% 以上進んだレーン」を優先し、重なりを防ぐ。

### D4: コメントデータは fetch ベースで外部化

**採用**: `CommentManager.load(url)` で JSON を fetch し、フォールバックは組み込みのデフォルトコメント配列。

```
今回: fetch('/comments.json')
将来: fetch(import.meta.env.VITE_COMMENTS_URL ?? '/comments.json')
```

**理由**: URL を変えるだけで Azure Blob Storage 等への差し替えが可能になる。
**代替案**: TypeScript 配列でハードコード — 差し替えに再デプロイが必要になるため却下。

### D5: コメント JSON のカテゴリ構造

```json
{
  "common": ["..."],
  "difficulty": {
    "easy": ["..."],
    "normal": ["..."],
    "hard": ["..."]
  },
  "events": {
    "stumble": ["..."],
    "goal": ["..."],
    "backPain": ["..."]
  }
}
```

ゲーム開始時に `common + difficulty[id]` をアクティブプールとして合成し、通常スポーンに使用する。
イベント発生時は `events[type]` からバーストスポーンし、アクティブプールも混在させる。

### D6: HUD トグルは HTML ボタン

**採用**: `pause-btn` と同様に HTML ボタン要素として `index.html` に追加し、`HUD` クラスで制御する。
**理由**: 既存の HUD が HTML オーバーレイ実装であり、一貫性を保てる。

## Risks / Trade-offs

- **フォント未ロード時のフォールバック** → DotGothic16 は Google Fonts で読み込み済み。ロード前に描画されても sans-serif にフォールバックするため致命的ではない。
- **コメント密度が高すぎる場合の視認性低下** → バースト数とスポーン間隔をパラメータ化しチューニング可能にする。
- **fetch 失敗時（オフライン等）** → フォールバックとして最小限のハードコードコメント配列を使用する。
- **モバイルでの文字サイズ** → Phaser の Scale.FIT により 960x540 キャンバスが拡大表示されるため、デザイン時の 24px が実機では適切なサイズになる。

## Migration Plan

- 既存コードへの破壊的変更なし
- `GameScene` に `CommentManager` を追加し、既存のイベントリスナーに便乗する形で統合する
- `public/comments.json` はデプロイアセットとして追加するだけでよく、既存の Vite ビルドに影響しない

# Design: smartphone-portrait-layout

## Context

確定申告ラン（Kakutei-Run）は Phaser 3 + TypeScript で構築された横スクロールアクションゲームである。現在のキャンバスサイズは 960×540（16:9）で、`Phaser.Scale.FIT` + `CENTER_BOTH` によって画面にフィットさせている。

スマートフォン縦持ち（例: iPhone 15 の 393×852）では、横幅に合わせてスケーリングされるため表示高さが約221pxとなり、画面の26%しか使えない。UIテキストもスケール率0.41で縮小され、視認性が著しく低い。

ターゲットユーザーの過半数が iOS Safari の縦持ちでプレイすることを想定しており、「ゲームロジックを維持しつつスマホ体験を改善する」ことが求められる。

既存のゲームプレイ領域（960×540）のロジック・スプライト座標・物理パラメータは安定しており、これを破壊せずにレイアウトを改善することが設計の中心的な制約である。

## Goals / Non-Goals

**Goals:**

- スマートフォン縦持ちで画面の 70% 以上を使用して表示する
- UIテキスト（タイトル・ボタン・HUD・リザルト）の視認性を大幅に改善する
- ゲームプレイの物理・座標・スクロール等のロジックに一切変更を加えない
- UI下部ゾーンでのタップがジャンプ操作として受け付けられる
- iOS Safari（アドレスバー表示あり）でも適切に表示される

**Non-Goals:**

- ゲームプレイ領域のアスペクト比変更やオブジェクト再配置
- PC向けの最適化（PCは左右余白あり、720×1080で表示される）
- 横画面（ランドスケープ）専用の別レイアウト対応
- レスポンシブな動的レイアウト切り替え（キャンバスサイズは固定）

## Decisions

### D1: キャンバスサイズを 960×1440（2:3）に変更する

**決定**: キャンバスを 960×1440 の縦長に変更し、Phaser の `Scale.FIT` + `CENTER_BOTH` を維持する

**理由**:

- 2:3 は iPhone（アドレスバー有り）〜 Android の縦画面に対して安全にフィットするアスペクト比
- 9:16（960×1700）だとアドレスバー有りの古い端末で上下が切れるリスクがある
- 横幅 960px を維持するため、ゲームプレイ領域の論理座標が一切変わらない
- iPhone 15 での表示: 393×589px（画面占有率 69%）、iPhone SE: 375×563px（占有率 84%）

**却下した選択肢**:

- 960×1700（9:16）: アドレスバー有り端末でのはみ出しリスク
- 540×960（横幅縮小）: ゲームオブジェクトの前方視認距離が 56% に減少し、全物理パラメータの再調整が必要
- 横幅540 + 縦960: ゲームロジックの座標系が変わり、改修コストが大きい

---

### D2: キャンバスを3ゾーンに分割する

**決定**: 上から UI上部（360px）・ゲームプレイ（540px）・UI下部（540px）の3ゾーン構成

```text
960 × 1440

┌──────────────────────────────────────┐ y=0
│          UI上部ゾーン (360px)        │
│  タイトル / HUD / ステータス         │
├━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┤ y=360
│          ゲームプレイゾーン (540px)   │
│          (既存ロジックそのまま)       │
├━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┤ y=900
│          UI下部ゾーン (540px)        │
│          (タップ→ジャンプ受付)       │
└──────────────────────────────────────┘ y=1440
```

**理由**:

- ゲームプレイゾーンが既存の 960×540 と完全一致するため、内部の物理座標を変更不要
- UI上部 360px はスケール後でも iPhone で約 147px 確保でき、HUD・タイトル等のフォントが十分読めるサイズ
- UI下部 540px はリザルト画面でスコア・ボタン配置に十分な領域、かつゲーム中はタップ受付領域として機能
- 上部360 + 下部540 = 900px あるため、UI要素を大きなフォントで余裕をもって配置できる

---

### D3: ゲームプレイゾーンのオフセット実装方式

**決定**: 各シーンでゲームオブジェクトの y 座標に `GAME_ZONE_Y = 360` のオフセットを加算する

**理由**:

- Phaser の Camera offset を使う方法もあるが、HUD（setScrollFactor(0)）やパーティクルとの整合性が複雑になる
- 定数オフセット加算の方がシンプルで、影響箇所が `GameScene.ts` と `ScrollManager.ts` に限定される
- `GROUND_Y`, `GROUND_HEIGHT` などの既存定数はゲームゾーン内のローカル座標として維持し、画面配置時に `+ GAME_ZONE_Y` するだけ

**却下した選択肢**:

- Phaser Camera scrollY: HUD の setScrollFactor(0) と干渉し、追加の補正が必要
- 別 Scene をオーバーレイ: Phaser の Scene overlay は入力イベントの伝播が複雑

---

### D4: タップ操作の受付範囲

**決定**: Phaser の `this.input` はキャンバス全域のポインターイベントを受けるため、特別な実装は不要。ゲームプレイ中は UI下部ゾーンのタップも自動的にジャンプ操作として受け付けられる

**理由**:

- 既存の `Player.setupInput()` で `this.input.on('pointerdown')` を使用しており、canvas 全体が受付範囲
- タイトル・リザルト画面ではボタンが `setInteractive()` されているため、Phaser のイベント伝播で自然に処理が分離される
- 追加実装が不要であり、バグの混入リスクがない

---

### D5: シーン別のゾーン使い分け

**決定**: 各シーンでゾーンの用途を以下のように定義する

| シーン       | UI上部 (0-360)              | ゲーム (360-900)   | UI下部 (900-1440)                           |
| ------------ | --------------------------- | ------------------ | ------------------------------------------- |
| タイトル     | タイトル文字 + サブテキスト | 背景装飾 or 空き   | 難易度選択ボタン（縦並び） + スタートボタン |
| ゲームプレイ | HUD（レシート数, 距離）     | 横スクロールゲーム | 空き（タップ→ジャンプ受付）                 |
| リザルト     | クリア/ゲームオーバー表示   | 演出（紙吹雪など） | スコア + リトライ + シェアボタン            |
| Boot         | ローディングバー全体        | —                  | —                                           |

**理由**:

- シーンごとに最適なレイアウトを定義することで、縦長画面を有効活用できる
- タイトル画面のボタンを縦並びにすることで、スマホでのタップ領域が十分に確保される

## Risks / Trade-offs

- **[PC表示での左右余白]** → PC（1920×1080）では 720×1080 で表示され、画面の37%しか使わない。スマホファーストの設計方針として許容する。PC向け最適化は Non-Goals として明示。
- **[ゲームゾーンのオフセット管理]** → y+360 のオフセットを正しく適用しないとオブジェクトがゾーン外に描画される。定数 `GAME_ZONE_Y` を一元管理し、`ScrollManager` と `GameScene` の2箇所のみで使用することで影響を局所化する。
- **[背景画像の再生成]** → プレースホルダー背景（bg_far, bg_near）の高さを540→1440に変更する必要がある。ただし `generate-placeholders.mjs` の数値変更のみで対応可能。
- **[既存テストへの影響]** → `MapGenerator.test.ts` 等でハードコードされている 960/540 はゲームプレイゾーンのローカル座標のため変更不要。ただし結合テストがある場合は画面サイズの変更を反映する必要がある。
